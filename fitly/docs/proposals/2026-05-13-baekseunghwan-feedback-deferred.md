### 발의안 — 문항 단위 crop + 페이지 연결 UI (#1·#2·#3)

#### 배경

검토자가 가장 강하게 지적한 영역. 본 PR 의 Lightbox 가 *학습 뷰어 약함* 문제(#5·#6)는 해결하지만, **이미지 안에 여러 문항이 함께 보이는 구조 자체**(#1·#2·#3) 는 현행 데이터 모델에서는 해결할 수 없다.

현재 `exam_items.stem_image_path` 는 *PDF 한 페이지 전체* 이미지를 가리킨다. 즉 한 페이지에 여러 문항이 있으면 같은 이미지가 여러 row 에 공유되며, 사용자는 화면에서 *현재 문항의 시작·끝* 을 직접 식별해야 한다.

#### 제안 — 시드 파이프라인 확장

1. **`exam_items` 스키마 확장**
   - `stem_bbox` (jsonb) — 문항이 속한 페이지 안 bounding box `{page: 1, x, y, w, h}`. 페이지 좌표는 PDF 원본 기준 (px).
   - `stem_page_range` (jsonb) — 문항이 여러 페이지에 걸친 경우 `[{page, bbox}, ...]` 배열.

2. **`docs/harness/seed-llm-prompts.md` step02 / step03 갱신**
   - LLM 에 *문항 시작 부호* (예: `1.`, `1번`, `[1]`) 위치를 보고하도록 prompt 추가
   - 다음 문항 시작 위치 - 1 까지를 현 문항 영역으로 산정

3. **(선택) 사용자 화면 client-side crop**
   - `<canvas>` 또는 CSS `clip-path` 로 큰 페이지 이미지에서 bbox 영역만 잘라 표시
   - 또는 시드 단계에서 `pdftocairo` 로 페이지 png 생성 후 `sharp` 로 bbox crop 하여 별도 png 저장

#### 영향

- 헌법 §16 (스코프 보호) 검토 — v3.5.3 §16 단서 *"외부 사용성 평가·접근성 점검·시각 위계 보강에 기반한 UI 개선은 기존 기능 다듬기로 분류"* 안에 들지 *애매*. crop 자체는 *데이터 모델 확장* 이라 본 단서 안에 들지 않을 수 있음.
- 단 검토자가 *"기출 분석 서비스에서 가장 중요한 순간은 사용자가 문제를 읽고 바로 풀 수 있는 순간"* 이라 강조한 점, 그리고 본 PR 의 Lightbox 가 *부분 해결*에 그친다는 점 — *별도 발의안*으로 헌법 §16 단서 *데이터 모델 확장 정합 단서 신설* 검토 가치.
