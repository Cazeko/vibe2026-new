## 발의 1 — 헌법 제2조 (제품 정의 · 공간 정책) 개정

### 1.1 동기 (제42조 1호)

외부 평가 (Page 17·총평 3번):

> *"이 서비스는 이동 중 학습(팟캐스트 등)을 표방하므로 모바일 웹(모바일 사파리/크롬)에서의 사용성이 70% 이상 중요합니다. LNB가 햄버거 메뉴로 빠지는지, 2단 스플릿 뷰가 모바일에서 상하 배치(스택 구조)로 잘 변환되는지 미디어 쿼리(Breakpoints) 설계가 철저해야 합니다."*

### 1.2 범위 (제42조 2호)

- `.claude/CONSTITUTION.md` 제2조 (제품 정의)
- `DESIGN.md` §1 (제품 컨텍스트) · §10.1 (브레이크포인트)
- `src/app/(main)/layout.tsx` (`hidden lg:block` 제거)
- `src/components/shared/tablet-gate.tsx` (제거 또는 안내 격하)
- 신규 컴포넌트: 모바일 햄버거 + drawer

### 1.3 사유 (제42조 3호)

현행 헌법 §2 / DESIGN §1.공간:

> *"태블릿 가로(1024×768) PWA · 데스크톱 함께 지원 · 모바일 폰은 폴드백"*

이 정책은 v1.9 시점 시연 우선 정책 (헌법 v1.9 제5조 단서). v3.0 컨셉 피벗 (편입 영어 → 초등 임용 + 팟캐스트) 이후에도 본문 유지. 그러나 **임용 수험생의 학습 행동 패턴은 통근·이동·짧은 휴식 시간 활용이 큰 비중** 차지 — 팟캐스트 트랙은 본래 모바일 1차. 외부 평가는 본 사용 시나리오를 정확히 짚었다.

### 1.4 영향 (제42조 4호)

**채택 시:**
- `(main)/layout.tsx` 의 `hidden lg:block` 제거 → 모바일에서도 메인 콘텐츠 노출
- 새 `MobileMenuButton` 컴포넌트 추가 (lg 미만 햄버거)
- `AppSidebar` 를 모바일 drawer 로 확장 (`fixed inset-y-0 -translate-x-full` + open state)
- 모든 페이지의 그리드를 `grid-cols-1` 기본 + `md:` `lg:` 단계로 점진 확장 (이미 대부분 정합)
- TabletGate 제거 또는 "권장 안내" 토스트 1회 노출로 격하
- 풀이 페이지 스플릿 뷰 (현재 max-w-3xl 단일 컬럼) 도 모바일 상하 스택 정합 검토

**비채택 시:**
- 모바일·태블릿 세로 사용자가 메인 콘텐츠 접근 불가 (현행 의도)
- 팟캐스트 트랙 사용성 외부 평가 미달 유지

### 1.5 승인 절차 (제42조 5호)

본 발의는 사용자(주관자) 명시 승인 후, 별도 PR 로 위 본문/코드 변경을 단행한다. 채택 동의 시 `v3.5.3` 또는 후속 omnibus 개정으로 `.claude/CHANGELOG.md` 갱신 동반.

## 발의 2 — DESIGN.md §9.1 (어조 · AI 추천 본문) 보강 — "동기부여 카피"

### 2.1 동기

외부 평가 (Page 1·1번):

> *"안녕하세요, 주인님."은 메이드 카페나 게임에서 쓸 법한 워딩입니다. 진지한 수험 앱에서는 "ㅇㅇㅇ님, 오늘의 목표를 향해 달려볼까요?" 등 동기부여형 워딩이 적합합니다."*

본 PR 의 P0-01 호칭 변경 ("선생님" 고정) 으로 "주인님" 워딩 우려는 해소. 다만 평가의 두 번째 의도 — **동기부여형 카피 적용** — 는 별도 사안.

### 2.2 범위

- `DESIGN.md` §9.1 어조 절 보강 (동기부여 카피 가이드)
- `src/components/shared/dashboard-header.tsx` subtitle 동기부여 톤 적용 검토
- AI 추천 카드 본문 (`ai-recommend.tsx`) 일부 보강

### 2.3 사유

헌법 §4 펀치라인 *"합격은 시간이 아니라 적합도다."* 의 *"적합도"* 톤은 **사실 + 자기 진척 인식** 강조. 동기부여 카피는 이 톤과 정합할 수 있으나 *"달려볼까요?"* 같은 표현은 자칫 강제·재촉 인상을 줄 위험.

### 2.4 영향

**채택 시 (보수적 적용 권고):**
- subtitle: `"오늘의 풀이·키워드·오답 트랙이 자동으로 채워지고, 추천 팟캐스트가 생성됩니다."` → `"오늘의 적합도 한 점, Fitly가 함께 짚어 드릴게요."` 류
- AI 추천 본문: 단정형 → 제안형 ("...해드릴까요?") 일부 보강 (이미 일부 정합)

**비채택 시:**
- 현행 사실형 톤 유지. *"학습 진척도·정답률 흐름"* 같은 데이터 중심 표현 그대로.

### 2.5 승인 절차

채택 동의 시 DESIGN §9.1 변경 + 위 컴포넌트 카피 변경을 별도 PR 로 단행. 비채택 시 본 발의안 archive.

## 발의 3 — 헌법 제16조 (스코프 보호) 단서 — "기존 기능 다듬기 vs 신규 기능" 경계 명확화

### 3.1 동기

P2 작업 중 다음 외부 평가 항목이 "기존 기능 다듬기"와 "신규 기능 추가" 사이 경계가 모호함:

- **PDF 이미지 zoom 컨트롤러 (인라인 +/- 버튼)**: 이미지 자체는 기존, zoom 컨트롤러는 신규 UI
- **풀이 페이지 스플릿 뷰 + 드래그 리사이저**: 본문/답안 분리는 기존, splitter 패턴은 신규
- **카드 hover Y축 이동 트랜스폼**: 일부 카드에 적용 사례 있음 (me 페이지 배지) — 일관 적용

### 3.2 범위

- `.claude/CONSTITUTION.md` 제16조 단서 추가
- 또는 시행규칙 (`rules/`) 으로 분리

### 3.3 사유

현행 §16: *"3주 대회 기간 내 신규 기능을 추가하지 아니한다. 모호한 요청은 기존 기능 다듬기로 해석한다."* — 다듬기 범위가 외부 평가 반영 시 자주 모호. 명확한 가이드라인 없으면 매번 사용자 결정 요청 발생.

### 3.4 영향

**채택 시 (단서 예시):**
- "기존 기능 다듬기" 정의: 사용자가 이미 도달 가능한 UI 의 가독성·접근성·시각 위계 개선
- "신규 기능 추가" 정의: 새 라우트·새 사용자 인터랙션 패턴 (drag/resize/zoom) 도입
- 경계 케이스는 사용자 즉시 발화로 결정

**비채택 시:**
- 현행 매번 사용자 결정 요청 패턴 유지

### 3.5 승인 절차

본 발의는 자체 우선순위 낮음. 발의 1·2 와 함께 검토 시점에 결정.
