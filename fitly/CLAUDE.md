# Fitly 헌법 (CLAUDE.md) — v3.0

> 본 문서는 Fitly 프로젝트의 최고 규범이며, 모든 코드·기획·발표·커뮤니케이션은 본 헌법에 종속된다.
> 본 헌법과 다른 모든 문서·합의·관행이 충돌할 경우 **본 헌법이 우선**한다.
> 단, 본 헌법은 사용자(주관자)의 명시적 승인이 있을 때에만 개정될 수 있다.

---

## 전문 (Preamble) [v3.0 재작성]

우리는 한국의 모든 초등교사 임용시험 준비생이 *지역·소득에 관계없이* **학원·인강처럼 체계적으로** 자신의 학습을 운영할 수 있게 한다.
사설 학원·인강이 주는 세 가지 — ① 강의 ② 자료 ③ 일정 — 중 **②③의 자동화에 집중**한다. 콘텐츠(강의)는 우리의 영역이 아니다.
공개된 24년치 임용 기출(국가공식)을 토대로, 우리는 **시드된 출제 트렌드**(영역·인지수준·문항형식·키워드)와 **3 트랙 SRS**(풀이/키워드/오답), 그리고 **NotebookLM 스타일 팟캐스트**로 사용자의 학습을 돕는다.
**합격 보장·합격 컷 추정 등 외부 데이터 없이 만들 수 없는 약속은 일체 하지 아니한다** (제3조의2 정합).
모든 결정은 사용자의 절박함을 출발점으로 삼되, *지킬 수 없는 약속을 만들지 않는 정직성*을 최상위 미덕으로 둔다.

## 제0장 · 최우선 규범 (절대 우선)

### 제0조 (호칭과 어투)
1. Claude는 사용자(주관자)를 **"주인님"**으로 호칭한다.
2. Claude는 모든 사용자 대화에서 **존댓말**만 사용한다. 반말·평어를 일체 금한다.
3. 본 조항은 본 헌법의 **모든 다른 조항보다 우선**한다. 다른 조항이 본 조항과 충돌할 시 본 조항을 따른다.
4. 코드·커밋 메시지·변수명에는 본 조항이 적용되지 아니한다. (제22조 2항과 양립)

## 제1장 · 정체성

### 제1조 (제품의 명칭)
본 제품의 정식 명칭은 **`Fitly` (한글: 핏리)** 이다. 그 외의 명칭(D-day, 디데이, NotebookLM++, StudyOS 등)은 일체 사용하지 아니한다.

### 제2조 (제품의 정의) [v3.0 재정의]
Fitly는 **한국 초등교사 임용시험(1차 — 교직논술 + 교육과정A·B)** 을 준비하는 자를 위한 **학습 플래너 PWA**이다. 24년치 공개 기출(국가공식)을 영역·인지수준·문항형식·키워드로 태깅한 시드 위에서, 사용자는 ① 출제 트렌드(히트맵·토픽맵·로드맵)로 약점을 발견하고, ② 서술형 풀이 / 키워드 / 오답의 **3 트랙 SRS**로 반복 학습하며, ③ NotebookLM 스타일 자동 생성 팟캐스트로 이동중에도 청취 학습한다.

### 제3조 (사명) [v3.0 재작성]
1. 사설 학원·인강을 끊을 수 없는 환경의 학생도 *체계적으로* 임용 학습을 운영할 수 있게 한다 (격차의 *완전 해소*가 아닌 *완화*임을 정직히 인정한다).
2. 공개된 임용 기출을 자동 태깅하여 출제 트렌드·풀이·키워드를 즉시 가시화한다 (시드 데이터).
3. 본인 풀이를 SRS 카드화 → 오답 합류 → 키워드 누적 흐름으로 변환한다.
4. 시험일까지의 *학습 일정*과 *진척도 추적*을 자동화하여 본인이 학습의 방향과 페이스를 결정할 수 있게 한다.

### 제3조의2 (포지셔닝 — v3.0 갱신)
1. Fitly 는 학원·인강의 *대체*가 아니라 **그 영역이 못 주는 자동화·플래닝의 보강** 도구이다.
2. 즉, 콘텐츠(강의)가 아닌 **자동화(시드 태깅·카드화) 와 플래닝(일정·진척)** 에 집중한다.
3. 합격 보장·합격 가능성·지역별 절대 적합도 산출 등 *외부 비공개 데이터에 의존하는 약속*은 일체 하지 아니한다.
4. 지역 교육청별 합격 컷·평균은 사실상 비공개이므로, 본 제품은 그것을 *추정으로 채우지 아니한다*. 사용자에게도 미보유 사실을 명시한다 (제29조 정합).
5. 우리의 정직성 자체가 사설 인강·학원과의 차별 메시지이며, 발표·UI·홍보에 일관되게 표명한다.

### 제4조 (펀치라인 — 절대 변경 금지)
> **"임용은 열심히 하는 게 아니라, 맞게(Fit) 하는 게임입니다."**

본 펀치라인은 발표 슬라이드 1번 및 클로징에 반드시 동일 문장으로 노출한다.

> *해설 (v3.0)*: 본 헌법에서 **"맞게(Fit)"** 의 의미는 "지역별 합격 적합도"가 아니라 **"본인 페이스에 맞는 학습 일정·진척"** 으로 해석한다. 즉, *내 시험일에 맞게, 내 약점 영역에 맞게, 내 망각 곡선에 맞게* 학습한다는 뜻이다.

## 제2장 · 핵심 원칙 (수상작 DNA)

### 제5조 (진입장벽 0)
사용자가 첫 가치를 경험하기까지의 행동 수는 **3 이하**여야 한다 (가입 → 시험일 등록 → 대시보드 진입).

> 단서 (v3.0): 첫 가치 경험은 *"시험일을 입력한 직후 24년치 출제 분포 히트맵이 즉시 보이고, 오늘의 3 트랙 SRS 큐가 자동으로 채워지며, 추천 팟캐스트 1편이 자동 생성되는 순간"* 으로 정의한다. 첫 자료 업로드는 선택 행동이며, 시드 데이터로 첫 가치는 충족된다.
>
> 단서 (v1.9): 본 대회 기간의 시연 디바이스는 **태블릿 가로(1024×768 기준) PWA**로 한다. 행동 수 3 이하 원칙은 유지하되 측정 디바이스가 모바일 폰에서 태블릿 가로로 이동한다.

### 제6조 (즉각적 가치)
모든 핵심 액션은 **5초 이내** 가시적 결과를 반환해야 한다 (사진 업로드 → 5초 내 시카드 변환 등).

### 제7조 (마법성)
사용자가 "이게 어떻게 됐지?" 라고 느끼는 변환 지점이 최소 1개 이상 존재해야 한다. 현재의 마법 지점은 다음 둘이다 (v3.0 갱신).
1. **시험일 입력 즉시 24년치 출제 트렌드 시각화 노출** (히트맵·토픽맵·로드맵 + 자동 채워진 3 트랙 SRS 큐)
2. **NotebookLM 스타일 팟캐스트 자동 생성** (영역·연도·주제 선택 → 5초 내 2인 화자 대화체 오디오 생성 시작)

### 제8조 (도파민 트리거)
사용자의 행동 직후 **숫자가 변하는** 시각적 피드백을 반드시 제공한다 (학습 진척도 +N% 애니메이션, 추출된 카드 수 카운트업, 연속 학습일 +1 등).

## 제3장 · 학습 진척도(Progress) 점수 [v3.0 전면 개정]

### 제9조 (Progress 공식)

```
Progress = (풀이 마스터율 × 0.5) + (키워드 마스터율 × 0.2) + (학습 일관성 × 0.3)
         (0~100 정규화)

풀이 마스터율    = FSRS state ≥ Review 인 quiz cards 수 / 전체 quiz cards 수
키워드 마스터율  = FSRS state ≥ Review 인 keyword cards 수 / 전체 keyword cards 수
학습 일관성     = 최근 14일 학습한 일수 / 14
```

본 공식은 `src/lib/progress/score.ts` 에 구현된다. 사용자 명시 승인 없이 계수·구조·해석을 변경할 수 없다.

본 점수는 **본인 학습의 상대 진척도** 지표이며, *지역별 합격 적합도와는 무관*하다.

> 오답 카드는 풀이의 again/hard 등급에서 자동 합류되며, 마스터되면 풀이 마스터율에 자연스럽게 흡수되어 별도 변수로 두지 아니한다 (중복 계산 방지).

### 제10조 (Progress 점수의 표현)
1. **"합격 가능성"**, **"합격 확률"**, **"점수 예측"**, **"지역 핏"**, **"교육청 핏"** 이라는 표현은 일체 사용을 금한다 (v3.0 — 임용 컨셉의 "지역 핏"·"교육청 핏" 도 금지어로 추가).
2. 정식 표현은 **"학습 진척도"**, **"Progress 점수"** 로 한정한다. UI·문서·발표 모두 이 두 용어만 사용한다.
3. UI 라벨에 "절대 점수가 아닌 본인 진척도" 를 명시한다 (예: "오늘의 학습 진척도 78 / 100 (본인 누적)").
4. 발표·UI·문서·홍보물에 위 1항을 위반한 표현이 발견되면 즉시 수정한다.

### 제11조 (데이터 토대 — v3.0 단순화)
1. Progress 점수는 **본인 계정의 학습 데이터**(`cards` 본인 FSRS 상태, `user_card_log`, `user_attempts`, `study_sessions`) 만으로 산출된다.
2. 외부 합격 컷·평균·가중치 시드에 의존하지 아니한다.
3. AI 가 합격 가능성·점수 예측을 추정해 노출하는 행위는 어떠한 경우에도 금한다.

### 제12조 (Progress 점수의 발표 방어 논리 — v3.0)
교수 심사위원의 "그래서 78점은 진짜인가요?" 질의에 대하여 다음 답변을 표준으로 한다.
> "본 78점은 *지역별 합격 적합도가 아닙니다*. 사용자 본인의 학습 데이터(풀이 마스터율 · 키워드 마스터율 · 학습 일관성) 3축의 가중 평균이며, 어제 70점에서 오늘 78점으로 오른 것은 *본인이 어제보다 8점만큼 더 잘 학습했다*는 의미입니다. 임용 합격 컷이 비공개라는 현실 제약을 정직하게 인정하고, 검증 불가능한 약속을 만들지 않기로 한 결정의 산물입니다."

## 제4장 · 구조와 범위

### 제13조 (사이드바 7+1+2 구조 — 변경 금지) [v3.0 전면 개정]
사용자 인터페이스는 **태블릿 가로(1024×768 기준) PWA 데시보드**로 한다. 좌측 고정 사이드바 + 우측 콘텐츠 그리드 구성을 표준으로 한다. 사이드바 메뉴는 다음 **Phase 1: 7개 + 신규 1개 (팟캐스트) + 시스템 2개**로 한정한다. Phase 2에서 3개(오답·학습분석·자료)를 추가한다 (신규 메뉴의 추가는 헌법 개정에 준한다).

**Phase 1 메인 메뉴 7 + 신규 1**:
1. **대시보드** (`/dashboard`) — 인사말 / KPI 4 (D-day·Progress·학습 시간·연속) / 24년치 영역 히트맵 미리보기 / 오늘의 3 트랙 큐 / 추천 팟캐스트 1편
2. **기출 분석** (`/exam-analysis`) — 4개 탭: ① 기출문제(연도×교시 색인) ② 분석(영역×연도 / 인지수준×연도 / 문항형식×연도 히트맵 3종) ③ 토픽맵(키워드 클라우드 + 반복 출제 키워드) ④ 로드맵(S/A/B/C 우선순위)
3. **풀이** (`/study/quiz`) — 연도×교시 → 문항별 답안 작성 + AI 모범답안 비교 + 자가 채점(FSRS grade)
4. **학습** (`/study`) — 3 트랙 SRS 큐: 풀이(QuizCard) / 키워드(KeywordCard) / 오답(MistakeCard)
5. **팟캐스트** (`/podcast`) — NotebookLM 스타일 자동 생성 라이브러리 + 영역·연도·주제 선택 + 청취 진척
6. **학습 계획** (`/study-plan`) — D-day 카운트다운 + 페이스 토글(가벼움/표준/집중). 일정은 SRS 자동, 사용자는 페이스만 선택
7. **마이 페이지** (`/me`) — 프로필·시험일·뱃지·라이브러리

**Phase 2 추가 메뉴 3**:
- **오답 노트** (`/mistakes`) — 풀이의 again/hard 자동 합류 뷰
- **학습 분석** (`/study-analysis`) — Progress 변화·영역별 마스터율·활동 히트맵
- **자료 관리** (`/materials`) — 사용자 PDF 업로드 + AI 자동 카드 추출

**시스템 2**:
- **설정** (`/settings`)
- **로그아웃** (액션)

> 모바일 폰 화면은 본 헌법 개정 이후 별도 반응형 후속 작업으로 다룬다. 본 대회 기간 시연 디바이스는 태블릿 가로로 한정한다.
>
> v1.6 시절의 모바일 3탭(홈/학습/내 오답)은 v1.9 개정으로 폐지되었으며, v3.0에서 사이드바 구조가 임용 컨셉에 맞춰 재편성됨.

### 제13조의2 (학습 카드의 출처와 분류) [v3.0 전면 개정]

본 헌법 제13조의 사이드바 메뉴 중 카드 노출 위치는 다음 세 종류로 구분되며, 명칭의 혼용을 금한다. 코드의 타입·변수·UI 라벨은 본 표를 따른다. v2.x의 `VocabCard` / `StudyCard` 명명은 **폐지**된다.

| 종류 | 영문명 | 출처 | `cards.type` | 노출 위치 (v3.0) |
|---|---|---|---|---|
| 풀이 카드 | `QuizCard` | 시드 `exam_items` 중 **2014~2026 서술형만** | `quiz` | 사이드바 "풀이" + 사이드바 "학습" 풀이 트랙 |
| 키워드 카드 | `KeywordCard` | 시드 `exam_items` 키워드(2002~2026 풀스코프, dedup) | `keyword` | 사이드바 "학습" 키워드 트랙 |
| 오답 카드 | `MistakeCard` | `QuizCard` 의 again/hard 자동 합류 + 사용자 직접 추가 | `mistake` | 사이드바 "학습" 오답 트랙 (Phase 2: 별도 "오답 노트") |

1. 세 종류는 모두 **`ts-fsrs` SRS 큐**에 통합되어 복습 일정이 산출된다.
2. 단일 사용자 시점에서 카드 출처를 가시적으로 표시하여 신뢰성을 보장한다 (예: 출처 배지 — "2024 교직논술 1번", "키워드 #2022개정 교육과정", "오답 합류").
3. AI에 의한 *무작위 학습 카드 생성*은 어떠한 출처에서도 금지한다 (제14조 3항 정합).
4. 데시보드의 파생 위젯(취약 영역 분석 / 학습 성과 추이 / 추천 팟캐스트)은 위 세 종류 카드의 *집계·시각화* 산출물이며 새로운 카드 종류로 보지 아니한다.
5. **카드 간 합류** — 사용자가 `/study` 에서 `QuizCard` 를 풀다가 "again" 또는 "hard" 등급으로 평가하면 동일 문항이 `MistakeCard` 로 자동 합류된다. 합류 시 `source_item_id` 로 출처 추적이 가능하며, 동일 문항의 중복 합류는 차단된다.
6. 시드 카드는 모든 사용자가 동일한 본문을 보지만, **FSRS 상태와 등급 로그는 사용자별 분리** (`user_card_log`, `user_attempts`).
7. v3.0 시점에서 `vocab_cards` / `study_cards` 테이블은 마이그레이션으로 `cards` (단일 다형 테이블, type 분기)로 통합한다.

### 제13조의3 (팟캐스트 콘텐츠 — v3.0 신설)

1. 팟캐스트는 **NotebookLM 스타일 2인 화자 대화체**로, Gemini multi-speaker TTS로 자동 생성된다.
2. `podcast_episodes` 의 scope 는 다음 두 가지로 한정한다.
   - `shared` — 시드 영역×연도·주제 단위로 자동 생성, 모든 사용자에게 공유
   - `user` — 사용자가 자기 약점 영역·기출 자료를 선택해 즉석 생성, 본인만 청취
3. 모든 팟캐스트는 본문에 **"AI 생성, 공식 해설이 아님"** 안내를 포함하며, UI 에 `verified: false` 배지를 표기한다 (제3조의2·제18조의2 정합).
4. 청취 진척은 `podcast_progress` 테이블로 사용자별 재개 가능하게 보존한다.
5. 외부 팟캐스트(YouTube·기타 사설)의 임베드는 본 헌법 v3.0 시점에 도입하지 아니한다 (저작권·정직성 리스크).

### 제14조 (모듈 구성 — Cut 결정)
다음 기능은 본 대회 출품 범위에서 **제외**되었으며, 다시 도입하려면 헌법 개정이 필요하다.
1. 마인드맵 (Mermaid·Markdown 모두 cut)
2. 자유 노트 에디터
3. 시카드의 *AI 무작위 생성* (RAG 추출만 허용)
4. 카톡 봇 진입점 (3주 리스크)

### 제15조 (지역 교육청 라벨 — v3.0 재정의)
1. 사용자는 응시 예정 **지역 교육청**을 선택 입력할 수 있다 (서울·경기·인천·부산·대구·광주·대전·울산·세종·강원·충북·충남·전북·전남·경북·경남·제주 17개).
2. 단, Fitly 는 어떠한 교육청에 대해서도 **합격 컷·평균·경쟁률 추정 시드를 보유하지 아니한다** (제3조의2 정합).
3. UI 에서 교육청별로 노출 가능한 정보는 **공개된 시험일·시험과목·시험시간·시험장소** 등으로 한정한다.
4. 본 조 입력은 선택 사항이며, 미입력 시에도 모든 학습 기능이 정상 동작한다 (시드 데이터는 전국 공통).

### 제16조 (스코프 보호)
본 대회 기간(3주) 내에는 본 헌법 제13~15조를 벗어난 기능을 추가하지 아니한다. 시간이 부족할 경우 *기능 추가*가 아닌 *기존 기능 다듬기*에 자원을 투입한다.

> 단서 (v1.9): 제13조 개정에 따라 추가되는 사이드바 7개 메뉴 영역(특히 학습 분석·기출 분석·마이 페이지·자료 관리)은 본 조의 '신규 기능'으로 보지 아니하며, 개정 후 기존 모듈의 재구성·확장으로 본다.
>
> 단서 (v3.0): 제13조 v3.0 개정에 따른 사이드바 재편(팟캐스트 신설, 풀이 분리, 학습 3 트랙 통합)은 본 조의 '신규 기능'으로 보지 아니하며, 임용 컨셉 피벗에 따른 재구성으로 본다.

### 제16조의2 (디자인 시스템 — DESIGN.md 우선) [v2.1 신설]
1. 모든 시각·UI 결정(폰트·색·여백·레이아웃·모션·컴포넌트 양식)은 `fitly/DESIGN.md` 의 **단일 소스 오브 트루스(Single Source of Truth)** 를 따른다.
2. 폰트·색·여백·레이아웃·모션의 어떠한 변경도 **사용자(주관자) 명시 승인** 없이 이루어지지 아니한다. 본 헌법 제16조의2의 변경은 헌법 개정에 준한다.
3. 메모러블 앵커는 **"공책 가로 괘선 + 세리프 헤드라인 + 학습 진척도만 컬러로 빛난다"** 이며, 이후 모든 시각 결정은 본 앵커를 *돕는가 거스르는가* 로 판단한다.
4. 액센트 컬러 `#1F5C4A` 는 *오직* (a) Progress 게이지 (b) 1차 액션 CTA (c) +N% 델타 (d) 활성 사이드바 항목 (e) AI 추천 카드 강조 — 5곳에만 등장한다. 그 외 컴포넌트 사용은 절대 금지한다.

### 제18조 (AI 사용 규정 — 모델 매트릭스)

   | 작업 | 모델 | 사유 |
   |---|---|---|
   | Vision OCR (시험지 사진 → 텍스트) | **`gemini-3.1-pro-preview`** | 현시점 최신·최고 성능 멀티모달 (1M 입력) |
   | 시카드/문제 구조화 추출 (텍스트 → JSON) | **`gemini-3.1-pro-preview`** | 긴 컨텍스트·구조화 출력 신뢰도 최상 |

4. PDF 파싱은 **`pdf-parse`** (서버) 및 **`gemini-3.1-pro-preview`** (이미지)을 표준으로 한다. 모델 ID는 `GEMINI_MODEL_PRO` 환경 변수로 관리한다 (제18조 5항 정합).

6. 본 매트릭스를 변경하려면 헌법 개정 절차를 거친다.

### 제18조의2 (AI 생성 정답·해설의 라벨 의무)
1. AI(Gemini)가 추정한 정답·해설은 사용자에게 노출될 때 반드시 **"검증 필요"** 배지와 함께 표시한다.
2. 사용자 신고·정정에 의해 검증된 카드는 다음 단계의 라벨로 승격된다 (제30조의2 참조).
3. 라벨 없는 AI 정답을 표시하는 것은 본 헌법 위반이며, 발견 시 즉시 정정한다.

## 제6장 · 개발 규율

### 제20조 (Read 먼저 Edit)
어떤 파일도 **Read 도구로 먼저 읽지 아니하고 수정하지 아니한다.** Edit/Write 도구는 Read 이후에만 호출한다.

### 제21조 (TypeScript Strict)
모든 코드는 `tsconfig.json`의 `"strict": true` 하에서 **타입 에러 0** 으로 빌드되어야 한다. `any` 사용 시 주석으로 사유를 명시한다.

### 제22조 (한국어 우선)
1. 사용자 가시 텍스트(UI 라벨, 에러 메시지, 알림)는 **한국어로 작성**한다.
2. 코드 내 식별자(변수·함수·타입명)는 **영어**로 작성한다.
3. 커밋 메시지는 한국어 또는 영어 자유.

### 제23조 (불필요 산출물 금지)
1. 사용자가 명시적으로 요구하지 않은 README·문서·데모 파일을 만들지 아니한다.
2. 사용자가 명시적으로 요구하지 않은 이모지를 코드·문서에 삽입하지 아니한다.
3. 사용자가 명시적으로 요구하지 않은 댓글·docstring·로깅을 추가하지 아니한다.

### 제24조 (검증 의무)
의존성 추가·스키마 변경·빌드 설정 수정 시 **반드시 `npm run build`** 또는 동등한 검증 명령으로 결과를 확인한다.

### 제24조의3 (검증의 운영 기준)
1. 다음 변경은 **`npx tsc --noEmit`** (5~10초) 검증으로 충분하다.
   - 컴포넌트·페이지의 단순 수정
   - 스타일·문자열·타입 보강
   - 기존 모듈 내부 리팩터링
2. 다음 변경은 **`npm run build`** (수 분)를 의무로 수행한다.
   - 의존성 추가·제거·업그레이드
   - DB 스키마 변경
   - `next.config.mjs` / `tailwind.config.ts` / `tsconfig.json` / `drizzle.config.ts` 수정
   - 미들웨어 변경
   - 라우트 추가·제거 (app/ 디렉토리 구조 변경)
   - 환경 변수 흐름 변경
3. 단위 테스트가 존재하는 영역(예: `src/lib/fit/score.ts`)을 수정한 경우 **`npm test`** 를 함께 수행한다.
4. 본 조항은 제24조의 *방법적 세분화*이며, 제24조의 의무를 약화시키지 아니한다.

### 제24조의4 (자동 푸시 기준)
1. Claude는 다음 시점에 사용자 추가 지시 없이 GitHub에 자동 커밋·푸시할 수 있다.
   - D-스텝(단위 작업)이 완료되고 검증(제24조의3 기준)이 통과한 직후
   - 헌법 개정이 적용된 직후 (개정 이력과 영향 범위가 같은 커밋 또는 직후 커밋에 포함될 것)
   - 보안 수정·명백한 버그 수정이 검증된 직후
   - 사용자가 명시적으로 "푸시"를 지시한 경우
2. 다음 시점에는 자동 푸시를 금한다.
   - 빌드 또는 테스트가 실패한 상태
   - 작업 도중의 미완 상태
   - `.env`, API 키, 토큰, 사용자 업로드 자료가 스테이지에 포함된 경우
   - 원격 저장소가 설정되지 아니한 경우 (사용자에게 URL 요청 후 보류)
3. 푸시 이전 의무 절차는 다음과 같다.
   - `git status`로 의도하지 않은 파일 포함 여부 확인
   - 비밀 정보(`.env*`, `*.key`, `*token*`) 스테이지 포함 여부 점검
   - **`git-commit-helper` 스킬을 호출하여 커밋 메시지를 표준화** (제25조 매핑)
4. `--no-verify`, `--force`, `--force-with-lease`, `main` 보호 브랜치에 대한 직접 푸시 등 위험 옵션은 **사용자 명시 승인 없이 사용하지 아니한다.** 훅 실패 시 우회하지 아니하고 근본 원인을 해결한다.
5. 푸시 직후 사용자에게 **커밋 SHA·메시지 요약·대상 브랜치**를 보고한다.
6. 본 조항은 제20조·제24조·제27~28조와 충돌하지 아니하며, 충돌 시 본 헌법의 보다 엄격한 조항이 우선한다.

### 제24조의2 (헌법 근거의 설명·진행 의무)
1. Claude는 사용자에게 진행 상황을 설명할 때 **본 헌법의 어떤 조항에 근거하여 진행했는지**를 함께 명시한다. 근거 명시 없는 보고는 정식 보고로 인정하지 아니한다.
2. Claude는 코드·기획·결정을 진행하기 전 **본 헌법의 어떤 조항에 근거하는지** 자기 검토하고, 명확한 근거가 없을 경우 작업을 보류한다.
3. 모든 결정은 적어도 하나의 조항에 근거를 둔다. 근거가 없는 결정은 임의적이며 본 헌법에 반한다.
4. 근거 조항이 모호하거나 누락된 경우, 제41조 (Claude의 개정 발의 의무)에 따라 개정 발의를 우선한다.
5. 본 의무는 단순한 보고 형식이 아니라, *결정의 사전 정당화 절차*로 기능한다.

## 제8장 · 보안 및 윤리

### 제27조 (저작권)
1. 학원·인강·사설 자체 교재의 무단 RAG 인덱싱을 금한다.
2. 모든 기출은 **국가공식 출처**(한국교육과정평가원 등)에서만 수집한다 (`fitly/kice_pdfs/`).
3. 출처는 데이터에 메타(`exam_papers.source_url`)로 보존한다.
4. AI 생성 모범답안·해설·팟캐스트는 공식 자료가 아님을 명시 라벨로 노출한다 (제18조의2 정합).

### 제28조 (개인정보)
1. 사용자 업로드 자료(사진·PDF)는 사용자 본인 외 접근을 차단한다 (Supabase RLS 의무). 단, 서버 사이드에서 Drizzle ORM이 PostgREST를 우회하여 직접 DB에 연결하는 경우, 코드에서 **모든 쿼리에 `user_id` 컬럼 일치를 강제**하는 것으로 RLS 의무를 갈음하며, 코드 리뷰로 누락 여부를 보장한다. RLS 정책 자체는 향후 PostgREST 경로 도입 대비로 유지한다.
2. 학습 기록·Fit 점수 히스토리는 사용자 동의 없이 외부 노출 금지.
3. 카카오 OAuth 사용 시 동의 항목을 최소화한다.

### 제29조 (책임 회피 표현)
1. **"합격 보장"**, **"100% 합격"**, **"합격 예측"** 등 결과를 단정하는 표현을 금한다.
2. 모든 점수·진단·추천에는 *참고용*임을 명시한다.

### 제30조 (할루시네이션 차단 의무)
1. 임용 시험·교육청 정보·합격 컷·합격 평균은 **반드시 시드 DB 또는 공식 출처**를 거쳐야 한다.
2. LLM이 임용 시험 메타 데이터를 자체 생성하도록 두지 아니한다.
3. 답변에 출처가 있는 경우 출처를 함께 표시한다.
4. AI 생성 모범답안·키워드·팟캐스트는 본문 시드(국가공식 기출)에 근거한 추론으로 한정하며, 외부 사설 학원의 해설 문구를 직접 인용·재생성하지 아니한다.

## 제9장 · 절대 금기 (Hard Don'ts)

### 제30조의2 (정답·해설의 출처 4계층 모델)
모든 카드의 정답·해설은 다음 4출처 중 하나로 명시 라벨링된다. 라벨 없는 정답·해설은 노출하지 아니한다.

| 라벨 | 영문 enum | 의미 | UI 배지 |
|---|---|---|---|
| 공식 | `official` | 학교 공시 정답 또는 검증 가능한 1차 출처 | "공식" (회색) |
| AI 추정 | `ai_estimate` | Gemini가 본문으로부터 추정한 답·해설 | "검증 필요" (앰버) |
| 자가 정정 | `user_self_corrected` | **단일 사용자(본인)** 가 직접 정정한 답·해설 | "내가 정정" (스카이) |
| 크라우드 검증 | `crowd_verified` | **2명 이상의 서로 다른 사용자가 동일한 답을 제출**하여 검증된 답 | "검증됨" (인디고) |

1. 단계적 결합 원칙 — 시점에 따라 다음 순서로 결합한다.
   - 단계 1: AI 추정으로 즉시 노출 (콜드 스타트)
   - 단계 2: 사용자 본인이 자기 카드를 정정한 경우 `user_self_corrected`로 변경 (단일 의견)
   - 단계 3: 학교별 공식 정답지가 확보되는 대로 `official` 승격
   - 단계 4: 동일 카드에 **2명 이상의 서로 다른 사용자가 동일 답을 제출**한 경우 `crowd_verified` 승격
2. `user_self_corrected`는 그 자체로 객관적 검증을 의미하지 아니하며, 본인의 자가 신뢰만을 표상한다. UI는 이 라벨을 "검증됨"과 시각적으로 구분되도록 표시해야 한다.
3. 라벨 변경 이력은 카드별 메타에 기록한다.
4. 발표·UI·홍보에서도 이 4계층을 신뢰성 방어 논리로 일관되게 사용한다 (제12조 정합).

### 제31조 (촌스러운 작명 금지)
1. "디데이", "D-day", "벼락치기", "족보", "한방 합격", "찐기출" 류 단어를 정식 명칭·UI에 사용하지 아니한다.
2. 사용자가 직접 채택한 표현(`Fitly`, `Progress 점수`, `학습 진척도`)을 표준으로 한다.

### 제32조 (족보·실명 데이터 금지)
1. 특정 교수·강사·학원의 *비공개* 출제 패턴 자료를 시드 DB에 인덱싱하지 아니한다.
2. 사용자 인터뷰에서 얻은 합격생 점수는 **익명화**하여 평균 산출에만 사용한다.

### 제33조 (벼락치기 프레임 금지)
1. UI·홍보·발표에 학습 윤리를 위배하는 인상의 표현(예: "벼락치기 캡슐", "마지막 7일 몰빵")을 사용하지 아니한다.
2. **"학습 적합도 향상"**, **"D-day 대비 학습"** 등 학습 효율 관점의 표현을 표준으로 한다.

### 제34조 (스코프 폭주 금지)
사용자 요청이 모호한 경우 새 기능을 추가하지 아니하고, 기존 3탭·핵심 모듈의 다듬기로 해석한다.

## 제12장 · 개정

### 제40조 (개정 요건)
1. 본 헌법은 **사용자(주관자)의 명시적 승인** 없이 변경하지 아니한다.
2. 어떠한 사소한 정정·표현 변경도 사용자 승인 없이는 적용하지 아니한다.
3. 기능·이름·공식·표현·금기 사항·스킬 매핑의 변경은 별도의 명시적 승인이 필요하다.

### 제41조 (Claude의 개정 발의 의무)
1. Claude는 다음의 경우 즉시 사용자에게 **개정 발의를 보고**해야 한다.
   - 본 헌법의 조항이 현재 작업·요청과 충돌하는 경우
   - 본 헌법에 누락된 영역이 발견되는 경우
   - 새 조항·금기·원칙의 추가가 필요하다고 판단되는 경우
   - 기존 조항이 더 이상 유효하지 않다고 판단되는 경우
2. 발의 의무는 *권고가 아닌 의무*이며, 발견 시점에 즉시 이행한다.
3. 발의 없이 작업을 진행한 결과로 헌법이 위반되었다면, 해당 작업은 무효이며 즉시 사용자에게 보고하여 처리 방침을 묻는다.

### 제42조 (개정 발의의 표준 형식)
Claude가 개정을 발의할 때는 반드시 아래 5요소를 사용자에게 제시한다.

1. **현행 조항** — 수정 또는 삭제 대상 조항을 그대로 인용
2. **제안 조항** — 신규·수정 후 문안을 그대로 작성
3. **개정 사유** — 왜 이 개정이 필요한지 1~3문장
4. **영향 범위** — 변경이 영향을 주는 코드·문서·기능·UI 목록
5. **승인 요청** — "승인하시겠습니까?" 문구로 명시적 답변 요청

### 제43조 (자가 개정의 금지)
1. Claude는 사용자의 명시적 승인 없이 본 헌법 파일(CLAUDE.md)을 직접 수정하지 아니한다.
2. 표현·문장의 사소한 정정도 반드시 사용자 승인을 거친 후 적용한다.
3. 본 헌법과 충돌하는 코드·문서·결정을 발견했을 때, 묵시적으로 우회하거나 임의 해석하지 아니하고 반드시 발의 절차를 거친다.
4. 사용자가 발의를 거부하거나 응답하지 않은 경우, 현행 헌법을 그대로 따른다.

### 제44조 (개정 필요성의 의무 점검 시점)
다음 시점마다 Claude는 헌법 개정 필요성을 의무적으로 점검하고, 필요 시 제41~42조에 따라 발의한다.

1. 새 모듈·기능의 추가 요청을 받았을 때
2. 기존 모듈·기능의 cut/축소가 검토될 때
3. 기술 스택·라이브러리·외부 서비스의 변경이 검토될 때
4. Gemini·외부 자료·심사위원 의견 등이 본 헌법과 상충할 때
5. 사용자가 "기존 결정과 다른 방향"을 명시하거나 시사할 때
6. 새로운 위험·백업 플랜·금기 사항이 발견될 때
7. 새 스킬이 추가되거나 작업 분류가 확장될 때

### 제45조 (개정의 기록)
모든 개정은 본 문서 하단의 「개정 이력」에 **일자·버전·사유·승인자(=사용자)** 와 함께 기록한다. 개정 이력에 누락된 변경은 효력이 없다.

- `.env.example` — 환경 변수 표준

### 부칙 제3조 (다음 단계) [v3.0 갱신]
본 헌법 v3.0 시행과 함께 다음 작업을 순차로 진행한다.
1. **시드 파이프라인 구현** — `kice_pdfs/` 정규화 → 텍스트 추출 → 문항 분리 → 영역/Bloom/형식/키워드 태깅 → 모범답안 생성 → 검토 → 카드 자동 파생
2. **DB 스키마 마이그레이션** — 기존 `vocab_cards` / `study_cards` / `mistakes` 폐지, `cards` (다형 단일 테이블) + `exam_papers` + `exam_items` + `podcast_*` 신설
3. **Phase 1 페이지 빌드 순서** — 기출분석 → 풀이 → 학습 → 팟캐스트 → 대시보드 → 학습계획 → 마이
4. **Phase 2 보류** — 오답 노트(별도) / 학습 분석 / 자료 관리
